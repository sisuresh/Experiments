## Notes — read carefully before planning

### This is a host/Soroban CAP, not consensus-layer
- The real implementation work is in **rs-soroban-env**: extend the SAC
  `transfer` and `mint` functions to accept a muxed-contract destination and
  emit it via the existing `to_muxed_id` event convention; prohibit
  `SC_ADDRESS_TYPE_MUXED_CONTRACT` in contract storage keys. **No new host
  functions** — reuse the existing `MuxedAddressObject` and its
  `get_address_from_muxed_address` / `get_id_from_muxed_address` accessors.
- **Exemplar (use it):** CAP-67 already did this exact muxing pattern for
  *accounts* (`MuxedEd25519Account`). Mirror its host implementation for
  contracts — find it in the local rs-soroban-env checkout
  (`grep -rn MuxedEd25519Account to_muxed_id soroban-env-host`). Scope:
  **destination-only**, **transfer/mint only** (`transfer_from` deferred),
  matching CAP-67. Reject any plan that adds source muxing or new host fns.

### stellar-xdr must be cloned first
- The canonical repo is not checked out at the target path. Clone it on `main`:
  `git clone https://github.com/stellar/stellar-xdr ~/dev/stellar-xdr`
- Author the new `SCAddress` arm on **`main`** behind the CAP `#ifdef`. The
  stellar-xdr PR targets `main`; `curr`/`next` regenerate automatically — do
  not edit them. Other repos vary by layer — see "Base branches" below.

### Horizon needs semantic event handling, not just regen
- Horizon parses SAC/contract events in `internal/ingest/contractevents/`. A
  muxed-contract destination changes what the transfer/mint event surfaces.
  `TestCoreLCMIngestion` only asserts ingestion doesn't error, so it will NOT
  catch a wrong/missing muxed id — implement the `contractevents` change and
  flag it for human review in the horizon PR.
